# 备份与恢复

插件升级通常不会触碰外部数据，但可靠恢复必须把“程序、最终连接器配置、项目素材、系统凭据”分开处理。备份前先解析**有效路径**，不要仅查看变量是否有值。

## 有效路径

| 项目 | 用户级变量已设置 | 用户级变量未设置 |
| --- | --- | --- |
| 状态根 | `IMAGE_CONTROL_STATE_ROOT` 的绝对路径 | `%LOCALAPPDATA%\CodexImageControl` |
| 项目根 | `IMAGE_CONTROL_PROJECTS_ROOT` 的绝对路径 | `<有效状态根>\data\projects` |

默认项目根位于状态根内部。删除整个默认状态根也会删除全部项目素材；只备份 `data\local` 不能保护这些项目。

在 Windows PowerShell 中可只读查看当前用户级设置：

```powershell
[Environment]::GetEnvironmentVariable("IMAGE_CONTROL_STATE_ROOT", "User")
[Environment]::GetEnvironmentVariable("IMAGE_CONTROL_PROJECTS_ROOT", "User")
```

空值必须按上表换算为默认路径。备份或删除前把两个最终绝对路径、是否嵌套、项目数量和总大小写入备份记录。

## 备份范围

必须保留：

1. 有效项目根中的项目 JSON、输入素材、正式图片和视频。
2. 有效状态根 `data\local\video-providers` 中经过人工审核的最终连接器资料：
   - `settings.json`；
   - 每个合法连接器目录的 `profile.json`；
   - profile 明确引用的 ComfyUI 工作流 JSON。
3. 当前正式版本号、发布 ZIP 的 SHA-256、两个有效根目录和自定义环境变量记录。

不要备份：

- 状态根的 `.runtime` 队列、锁、能力文件、临时文件和进程信息；
- `data\local\video-provider-setups` 中的接入草稿和上传文件；
- `data\local\backups\project-migrations` 中已由正式项目备份覆盖的旧项目 JSON 副本；
- Codex 的 `plugins\cache`、源码仓库、`node_modules`、`app\dist`、`server\dist` 或 `artifacts`；
- Windows 凭据库的明文导出。

`data\local` 即使没有密码，也可能包含私有端点、内部主机名和工作流。备份介质必须加密并限制访问，不能放进源码仓库、插件目录、公开云盘或 GitHub Release。

## 创建备份

1. 等待视频任务结束或明确取消，确认项目中没有活动请求。
2. 完全退出 Codex 和独立工作台。
3. 解析并记录有效状态根和项目根，确认它们不是磁盘根、用户主目录、源码仓库、插件缓存或共享目录。
4. 从磁盘根到目标逐级检查，不得存在符号链接、目录联接、挂载点或其他重解析点。备份工具必须采用“不跟随链接”模式；遇到任何链接条目直接停止并人工处理。
5. 把完整项目根和上述连接器允许列表复制到两个独立备份位置，其中至少一个不与工作电脑处于同一磁盘。
6. 为每份备份建立 `相对路径、文件长度、SHA-256` 清单，并记录文件数量、总大小、日期和插件版本。
7. 对照源与备份清单；随机打开项目 JSON、代表性图片、视频、profile 和工作流，确认不是空目录、损坏副本或链接目标的意外副本。

不要在应用仍写入时复制项目状态，不要把备份直接覆盖到已有目标目录，也不要让备份程序遍历 reparse/symlink 指向的其他位置。

## 恢复或换机

1. 在目标电脑安装 Windows x64 依赖、Node.js 22、Codex，以及视频功能所需的 ffmpeg/ffprobe。
2. 安装与备份记录一致的正式插件版本；先不要创建同名项目或连接器。
3. 选择两个经过核验的目录。状态根必须是新的普通目录；项目根可以是新的独立目录，或恰好为 `<状态根>\data\projects`。除此之外两者不得重叠，也不得位于源码、插件缓存、旧备份或共享目录内。
4. 确认目标路径的每个现有组件都不是重解析点，并且目标 `data\local` 和项目目标均为空、没有同名冲突。
5. 先恢复项目并核对项目清单，再只恢复最终连接器允许列表；不要恢复 `.runtime`、setup 草稿、迁移备份或其他 `data\local` 内容。
6. 启动插件，核对项目数量、名称、素材、图片和视频。
7. 人工审核每个连接器的 profile、工作流哈希、提交地址、状态地址和下载来源。
8. 需要凭据时，**只通过本机连接器界面重新录入**。不要手工创建 Windows 凭据库账户。
9. 逐个执行免费连接探测，不直接批量提交任务。

目标已有任何数据时停止恢复，另建空目录并生成冲突清单；不要覆盖同名项目或 profile。旧版本、旧电脑和原备份至少保留到一次完整恢复演练通过。

## 凭据恢复矩阵

文件备份不包含密钥，但 Windows 凭据库独立于状态目录，因此恢复后的实际表现取决于设备和 profile 安全范围：

| 场景 | 处理方式 |
| --- | --- |
| 同机同 Windows 用户，且已存在相同 profile ID 与相同 scope 的 scoped credential | 可能继续显示凭据已配置；先核对端点和范围，不确定时在 UI 清除后重录 |
| 旧 profile 只有无 scope 的 legacy credential | 不自动绑定；必须在 UI 重新录入 |
| 端点、认证方式、状态来源或下载认证范围变化 | 原凭据失效；必须在 UI 重新录入 |
| 新电脑或新 Windows 用户 | 必须在 UI 重新录入 |

怀疑旧配置或备份曾泄露时，先在服务端撤销或轮换密钥，再处理本机恢复。删除文件或恢复旧 profile 不能撤销服务端凭据。

## 恢复演练

至少在重要版本发布后或每季度做一次恢复演练：恢复到新的非重解析目录，对比文件数量和 SHA-256，打开代表性项目和媒体，确认缺少凭据时系统失败关闭，并确认不会出现维护者、旧项目或其他状态根的隐藏端点。
